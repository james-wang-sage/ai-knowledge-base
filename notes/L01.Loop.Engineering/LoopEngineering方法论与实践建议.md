# Loop Engineering：新 Agent 从零到上线的方法论与实践建议

> 这不是对课件逐页复述，而是结合整套课程内容后，提炼出的 Agent 项目启动方法，并加入对课件观点的工程化修正。

## 一句话结论

Agent 项目不应从 Prompt、模型或框架开始，而应从**用户目标、外部验收条件和现实副作用**开始：让模型处理语义与开放判断，让确定性代码控制权限、不变量、精确参数、状态转移和停止条件。

可以把它概括为：

> **目标与验收先行，模型负责思考，代码负责约束，外部事实负责证明。**

---

## 一、先判断：这个问题真的需要 Agent 吗？

在选择模型或 Agent 框架前，先用两条轴判断任务。

| 任务类型 | 单步 | 长程、多步骤 |
|---|---|---|
| 内容型 | 普通 LLM 调用 | 轻量 Agent 或工作流 |
| 执行型 | 工具调用 + 结果校验 | 完整 Loop Engineering |

- **内容型**：搜索、总结、分析、写作，错误通常可以修改或重试。
- **执行型**：写数据库、创建订单、发消息、支付，会改变外部世界。
- **单步**：一次模型或工具调用即可完成。
- **长程**：步骤之间存在依赖，需要多轮观察、更新和恢复。

执行型且长程的任务，最值得投入 DAG、机械状态、HITL、恢复和 Trace。单步小事不应套上重型 Agent 架构。

### 我的建议：先证明“非 Agent 不可”

如果普通函数、表单、搜索接口或固定工作流已经能够可靠解决问题，就不要为了使用 Agent 而加入自主循环。Agent 的引入应当解决模糊理解、动态决策或开放环境问题，而不是替代本来就确定的程序逻辑。

---

## 二、不要先写 Prompt，先走通用户旅程

从一句真实委托开始，回答：

> 用户最终想获得什么结果，而不是系统想调用什么工具？

先画出端到端旅程：

```text
用户委托
→ 理解目标
→ 补齐必要信息
→ 形成计划
→ 执行步骤
→ 确认高风险动作
→ 验证现实结果
→ 交付结果与回执
```

工程师应亲自手工走一遍现有业务流程，确认：

- 涉及哪些系统和权限；
- 精确参数分别来自哪里；
- 哪些操作不可逆；
- 哪些步骤可以安全重试；
- 什么才算真实成功；
- 失败以后能够回到哪里。

没有亲自走通数据库、订单、支付或外发流程，就不应急着让 Agent 自动执行它。

---

## 三、先写四份契约，再选择模型

### 1. 输入契约

- 用户最少需要提供哪些信息？
- 哪些字段可以推断，哪些必须追问？
- 字段的类型、范围和有效期是什么？

### 2. 输出契约

- 最终交付的是文字、结构化数据、文件，还是外部系统状态？
- 用户需要看到哪些证据和回执？

### 3. 成功契约

成功必须由外部事实证明，例如：

- API 返回成功，并且重新查询能找到记录；
- 文件存在且通过格式或内容校验；
- 数据库事务已经提交；
- 用户明确批准了不可逆操作；
- 测试由红变绿。

“模型说完成了”不构成成功。

### 4. 停止契约

提前定义：

- 成功时何时停止；
- 信息不足时何时询问；
- 高风险动作前何时阻塞；
- 一类错误最多重试几次；
- 时间、token 和费用达到多少时停止；
- 什么情况下转交人工。

边界和停止条件不是上线前补充的配置，而是 Agent 设计的起点。

---

## 四、建立“确定性外壳，有限自主内核”

课件的核心定律是：

> Agent 思考，机械平面执行。

这条原则正确，但不应被理解为所有计划都必须静态写死。更合适的分工是：

### 交给模型

- 理解模糊需求；
- 识别意图和抽取槽位；
- 生成候选计划；
- 分析工具观察结果；
- 在受限选项中做语义判断；
- 生成用户可读的说明。

### 交给确定性代码

- 身份、权限和工具白名单；
- 精确 ID、金额、路径和账户；
- 业务不变量和依赖约束；
- 状态转移；
- 重试与预算上限；
- 高风险审批；
- 最终成功判定。

### 我的修正：骨架固定，局部动态

“计划写进代码”不应等于把所有步骤都固化成静态 DAG。推荐：

```text
确定性入口
→ 模型生成或调整局部计划
→ 受约束执行
→ 确定性风险门
→ 外部结果验证
```

代码钉死不变量、风险边界和验收条件；模型保留解决开放问题所需的自由度。否则系统很容易退化成传统 BPM 工作流加一个自然语言入口。

---

## 五、把自然语言收敛成控制信号

入口不应把用户原话直接交给一个万能 Agent，而应先转成结构化信号：

```json
{
  "intent": "create_order",
  "confidence": 0.93,
  "slots": {
    "party_size": 2,
    "budget": 150,
    "time": "tonight",
    "category": "hotpot"
  }
}
```

设计要求：

- 意图来自有限枚举；
- 关键槽位有类型和约束；
- 低置信度进入澄清流程；
- 无法识别时统一进入 `unknown`；
- 路由程序消费结构化信号，不自行解析自由文本。

模型负责把“活的语言”收敛成“可控信号”，程序据此进行确定性路由。

---

## 六、根据任务拓扑选择编排方式

DAG 很重要，但不是所有 Agent 的默认答案。

| 场景 | 推荐结构 |
|---|---|
| 固定业务流程 | 状态机或 DAG |
| 信息检索与分析 | 有界 ReAct |
| 开放式编码任务 | Planner-Executor + 动态重规划 |
| 多个独立研究方向 | 并行 fan-out/fan-in |
| 长期运行任务 | 事件驱动工作流 |
| 反复优化产物 | Generate-Critique-Revise |

当任务适合 DAG 时，每个节点至少定义：

- 前置条件；
- 标准输入和输出；
- 可调用工具；
- 成功条件；
- 失败分类；
- 是否可重试；
- 补偿或回滚动作；
- 超时与重试上限；
- 是否需要人工审批。

调度器负责推进依赖关系，不代替模型做语义推理。

---

## 七、把能力拆成原子工具

好的工具应当小、明确、可验证：

```text
search_stores(category, location)
check_inventory(store_id, sku_id)
create_cart(store_id, sku_id, quantity)
create_order(cart_id, idempotency_key)
verify_order(order_id)
```

避免把所有能力包装成 `handle_everything(user_request)` 之类的黑箱。

每个副作用工具应声明：

- 严格输入 Schema；
- 结构化输出；
- 所需权限；
- 是否幂等及幂等键；
- 超时后的查证方式；
- 可重试错误和永久错误；
- 事务或补偿语义；
- 可验证的成功回执；
- 审计信息。

### 我的补充：重试前先查证

执行型 Agent 最危险的错误之一，是超时后盲目重试，从而重复下单、发送或扣款。正确原则是：

> 不确定上一次是否成功时，先查询外部状态，再决定是否重试。

---

## 八、状态逻辑分离，不必一开始物理分裂

课程将状态分成三个平面：

1. **叙事状态**：用户要什么、已经做到哪里；
2. **机械状态**：精确参数是什么、来自哪里；
3. **调度状态**：哪个步骤应该运行、是否完成或阻塞。

这个语义划分非常有价值，但小型 Agent 不必一开始就建设三套独立系统。可以先使用一个类型化状态对象：

```python
class AgentState:
    goal: Goal
    plan: list[Step]
    artifacts: dict[str, Artifact]
    tool_values: dict[str, ProvenancedValue]
    approvals: list[Approval]
    execution: ExecutionStatus
```

关键约束是：

- 精确值不能从叙事摘要反推；
- 调度状态不能由模型自由改写；
- 每类状态有明确写入者；
- 状态变化必须可以追踪。

先逻辑分离，等规模、团队或并发要求出现后，再物理分离。

---

## 九、把 Provenance 扩展到所有关键事实

Provenance 不应只服务于 `order_id` 等机械参数，还应覆盖事实、推断和生成内容：

```json
{
  "value": "...",
  "kind": "observed",
  "source": "tool-or-user",
  "producer": "search_tool",
  "created_at": "2026-07-11T12:00:00Z",
  "expires_at": "2026-07-12T12:00:00Z",
  "confidence": 0.98
}
```

至少区分：

- `observed`：从用户、工具或环境直接获得；
- `inferred`：模型根据证据推断；
- `generated`：模型新生成的内容。

研究、合规和知识库 Agent 中，最大的风险往往不是 ID 抄错，而是把模型推断误当成来源事实。

---

## 十、将软护栏升级为动作前硬校验

建议按照风险逐步开放权限：

```text
只读查询
→ 草稿生成
→ 可逆写入
→ 外发消息
→ 不可逆操作
→ 资金动作
```

高风险动作采用：

```text
生成不可变动作提案
→ 阻塞
→ 展示精确预览
→ 用户确认
→ 执行
→ 验证真实回执
```

有效审批必须说明：

- 即将执行什么动作；
- 使用哪些精确参数；
- 影响范围和不可逆后果；
- 为什么需要执行；
- 是否存在更安全的替代方案。

用户批准后，如果金额、对象或其他关键参数发生变化，必须重新审批。

### 我的补充：先做威胁模型

在开放工具前，还应回答：

- 谁可以发起任务，可能冒充谁？
- 模型能访问哪些数据？
- 网页、邮件和文档中的 Prompt Injection 能否影响工具调用？
- 工具被滥用时的最大损失是多少？
- 数据是否会跨用户、会话或租户泄漏？
- 工具返回值是否可信？
- Trace 是否会记录 token、密码或个人信息？

外部内容是数据，不是可信指令；数据平面与控制平面必须保持信任边界。

---

## 十一、验证强度应当与风险匹配

课件强调“验证必须来自循环之外”，方向正确，但不应被理解为模型评价没有价值。

模型自评适合：

- 发现格式和明显逻辑问题；
- 对候选结果排序；
- 做低成本初筛；
- 生成测试用例；
- 检查是否遗漏用户要求。

但在高风险场景中，它不能成为唯一证据。推荐使用验证梯度：

1. Schema 与静态规则；
2. 确定性测试；
3. 工具、数据库或环境真实状态；
4. 模型自评；
5. 独立模型或多样化评审；
6. 人工验收。

需要始终区分：

```text
tool_called != operation_succeeded
operation_succeeded != business_goal_achieved
```

自我评价可以成为证据，但不能在高风险场景中自我认证。

---

## 十二、评估整条轨迹，而不只看最终回答

Agent 的质量单位应从 response 转向 trajectory。至少评估：

- 是否正确理解意图；
- 是否选择了合适的流程；
- 是否遗漏或跨越步骤；
- 参数来源是否正确；
- 是否虚假报告成功；
- 是否触发了必要审批；
- 失败后是否正确恢复；
- 是否在成本和时间预算内完成；
- 最终结果是否通过外部验证。

推荐指标包括：

- 任务完成率；
- 虚假成功率；
- 参数绑定错误率；
- 人工介入率；
- 平均循环轮数；
- 工具失败恢复率；
- 单次任务成本与延迟；
- 高风险动作违规率。

评测集不能只有黄金成功路径，还应包括：

- 工具超时和脏数据；
- 模糊或恶意输入；
- 用户中途修改目标；
- 审批后参数发生变化；
- 重复请求；
- Prompt Injection；
- 运行中进程重启；
- 模型或 Prompt 升级后的回归测试。

---

## 十三、可观测性应是结构化运行记录

“玻璃罩”不应只是保存完整对话或展示漂亮 Trace。每次运行建议记录：

- `run_id` 和 `step_id`；
- 模型、Prompt、Skill 和工具版本；
- 状态转移；
- 工具名称和脱敏参数摘要；
- 重试次数与原因；
- token、费用和延迟；
- 审批人及批准的动作提案；
- 验证结果；
- 最终失败分类。

同时必须对秘密、token、密码和个人信息进行脱敏。可观测系统本身也可能成为数据泄漏源。

---

## 十四、按风险逐步开放，而不是一步到全自治

推荐的上线节奏：

### 阶段一：只读闭环

完成查询、比较和建议，不修改外部状态。

### 阶段二：草稿闭环

生成订单、消息或变更草稿，由人工执行。

### 阶段三：受控写入

允许低风险、可逆、幂等的写操作。

### 阶段四：长程恢复

增加持久化状态、断点恢复、失败补偿和结构化 Trace。

### 阶段五：高风险审批

接入支付、退款、外发等不可逆动作，同时保留 HITL。

只有前一阶段通过真实轨迹评测，才开放下一层权限。

---

## 十五、推荐的最小可行架构

不要一开始就堆叠多个 Agent、Reviewer、状态数据库和复杂 DAG。一个新 Agent 可以从下面的最小闭环开始：

```text
结构化入口
→ 有界 Agent 循环
→ 少量类型化原子工具
→ 单一持久化状态
→ 副作用动作前硬校验
→ 外部结果验证
→ 结构化 Trace
```

只有真实运行暴露出跨步、参数漂移、恢复困难、上下文膨胀或高风险问题时，再逐步加入：

- 静态或动态 DAG；
- 独立状态平面；
- Reviewer；
- 多 Agent 协作；
- 复杂补偿机制。

好的 Agent 不是拥有最多组件，而是以最少机制把当前风险降到可接受范围。

---

## 十六、新 Agent 开工清单

### 场景与目标

- [ ] 这个任务属于内容型还是执行型？
- [ ] 它是单步还是长程？
- [ ] 普通程序或固定工作流是否已经足够？
- [ ] 用户最终想获得的现实结果是什么？
- [ ] 成功由什么外部事实证明？

### 控制与状态

- [ ] 哪些判断交给模型，哪些规则必须由代码执行？
- [ ] 应采用 DAG、状态机、ReAct 还是其他拓扑？
- [ ] 精确参数保存在哪里？
- [ ] 参数、事实和推断是否具有 Provenance？
- [ ] 调度状态是否禁止模型直接改写？

### 工具与副作用

- [ ] 每个工具是否具有严格输入输出契约？
- [ ] 副作用工具是否幂等？
- [ ] 超时以后能否先查询再决定重试？
- [ ] 是否存在事务、回滚或补偿动作？
- [ ] 哪些动作需要不可变提案和 HITL？

### 安全与验证

- [ ] 是否分析了 Prompt Injection、越权和数据泄漏？
- [ ] 是否设置时间、成本、轮数和权限上限？
- [ ] 是否区分工具调用、操作成功和业务完成？
- [ ] 高风险结果是否有模型之外的验证？
- [ ] Trace 是否结构化且经过脱敏？

### 评测与上线

- [ ] 是否测试成功、失败、恶意输入和恢复路径？
- [ ] 是否保存可重放的真实失败轨迹？
- [ ] 是否有模型、Prompt、Skill 和工具版本信息？
- [ ] 第一版能否先以只读或草稿模式上线？
- [ ] 新增复杂机制是否由真实失败数据驱动？

---

## 最终方法

新 Agent 的合理工作顺序是：

> **判断是否需要 Agent → 走通用户旅程 → 定义输入、输出、成功与停止契约 → 划分模型和代码的职责 → 选择任务拓扑 → 原子化工具 → 建立类型化状态与 Provenance → 加入幂等、硬护栏和 HITL → 用外部事实验证 → 评估整条轨迹 → 按真实风险逐步开放权限。**

课件最有价值的提醒是：不要只调模型，要工程化整个循环。

需要进一步补上的工程判断是：不要因为开始工程化循环，就默认需要最重的架构。**确定性应覆盖风险，而不是消灭智能；复杂度应由真实失败驱动，而不是由架构想象驱动。**
